# LoopAnalyzer 现状分析报告（2026-03-02，已更新）

## 1. 当前定位

- 目标文件：`LoopAnalyzer.py`
- 当前策略：**仅保留非矩阵模式的稳定向量化分析**
- 已移除：矩阵相关识别与生成路径（`MATRIX` 分支）

本次调整的目标是先收敛主链路稳定性，再按阶段恢复高级功能。

---

## 2. 当前实现作用（基线能力）

`LoopAnalyzer.py` 当前负责将 `LoopInfo` 归一化为 `AnalyzedLoop`，并服务于后续 SVE 代码生成，包含：

1. 模式分类：`ELEMENTWISE` / `REDUCTION` / `CONDITIONAL` / `UNKNOWN`
2. 可向量化性规则检查：步长、简化依赖、条件分支控制变量修改
3. 类型推断：`type_context` + `c_type` 到 `DataType`
4. SVE 元信息映射：元素类型、后缀、`vl_hint`、`whilelt_suffix`
5. 批量入口：`analyze_loops()` 只分析最内层循环

---

## 3. 已完成的收敛改动

1. 从 `LoopPattern` 中移除 `MATRIX` 模式。
2. 删除矩阵识别辅助逻辑（如最深层矩阵特征判定）。
3. 恢复统一入口策略：只分析最内层循环。
4. 下游同步：去除矩阵模板分发与矩阵展开策略，报告统计回归到单一可向量化状态。

> 结果：当前主流程行为更一致，测试可稳定通过。

---

## 4. 仍需关注的问题（与矩阵无关）

### P1（建议优先）

1. 依赖判定仍偏启发式：
   - RAW/别名规则主要依赖 `index_offset` 与数组名。
   - 对复杂下标表达式、执行方向和跨语句关系覆盖不足。

2. 条件分支控制变量修改检测仍基于字符串匹配：
   - `startswith` 方法在 `++i`、`i+=2` 等形式下存在漏判风险。

### P2（健壮性）

3. 类型推断对复杂 C 声明修饰符兼容有限（`const/restrict/volatile` 等）。
4. 归约初值未结合归约算子（如乘法幺元应为 `1`）。

---

## 5. 保留后续扩展能力（关键）

虽然矩阵模块已下线，但建议保留以下扩展接口思想，方便后续逐步恢复：

1. **模式扩展点保留在 `_classify_pattern()`**
   - 后续新增模式时，继续采用“按优先级分类 + UNKNOWN 兜底”结构。

2. **统一输出结构 `AnalyzedLoop` 不变**
   - 后续新增模式应尽量复用已有字段，避免破坏下游接口。

3. **入口策略可配置化（后续再做）**
   - 当前固定“最内层优先”；后续可加参数切换：`innermost-only` / `outer-aware`。

4. **状态机预留**
   - `VectorizabilityStatus.PARTIAL` 可暂时保留，用于未来“部分可向量化”场景。

5. **代码生成分发可插拔**
   - 未来恢复矩阵时，建议以独立生成器模块回接，避免再次耦合主路径。

---

## 6. 分步恢复建议（后续）

1. 第一步：先做 AST 级依赖判定增强（不引入新模式）。
2. 第二步：加“可配置入口策略”，并在分析阶段输出更多上下文信息。
3. 第三步：以插件化方式恢复矩阵模式（分类 + 生成 + 测试同时上线）。

---

## 7. 总结

当前版本已经回到稳定、可维护的“非矩阵基线”。该基线保留了后续扩展的结构空间，适合按“先判定精度、再入口策略、后高级模式”的顺序逐步演进，降低回归风险。
